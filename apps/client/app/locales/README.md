# App Locales Directory

## Purpose

This directory contains translation files that are **bundled at build time** for performance optimization.

## Why This Directory Exists

### Build-Time Import Limitation
Next.js does not allow direct ES6 imports from the `public/` folder during build time. Files in `public/` are meant to be served as static assets via URLs at runtime, not imported as modules.

### Performance Optimization Strategy
To eliminate the initial HTTP request delay for English translations (50-200ms), we:

1. **Bundle English inline**: Import `en.json` directly into the TranslationProvider component
2. **Keep it in `app/locales/`**: This location allows TypeScript/Webpack to import the JSON at build time
3. **Use `partialBundledLanguages`**: i18next feature that mixes bundled and backend-loaded languages

### File Layout
- **`app/locales/en.json`**: The **single source of truth** for English strings. Imported at build time and bundled into the JavaScript bundle — this is what English users actually see. Add/edit UI strings here.
- **`public/locales/<lang>.json`**: All other languages, loaded on-demand via HTTP at runtime. Generated from the English base by the translation-sync script (see below). There is intentionally **no** `public/locales/en.json` — English is never loaded over HTTP (it comes from the bundle), so a second English copy would only be a drift hazard.

**Important**: when you add or change English strings in `app/locales/en.json`, run the translation-sync workflow below to propagate the new keys to the other languages. No manual copying step is needed.

## Translation Sync Workflow

Other-language files are generated by `packages/scripts/syncTranslations.ts` (`translations:sync`). It reads `app/locales/en.json` as the base, finds keys missing from each `public/locales/<lang>.json`, translates them with the configured **operations model** (resolved from the database — including the system LLM API keys), and writes them back. Keys present in a target file but absent from the base are pruned.

Run it against a deployed stage. The script resolves the MongoDB connection
string from the SST-linked `Resource.MONGODB_URI` for the current stage, so a
plain `sst shell` is enough — no env-var bridging required:

```bash
# Stage 'dev' = the staging environment (see root CLAUDE.md on stage naming)
./for-env dev pnpm sst shell --stage dev -- \
  pnpm --filter @bike4mind/scripts translations:sync
```

Prefer testing against a PR's own preview stage (`pr<N>`) rather than staging —
it hits the PR's environment and the connection is read-only (it only reads the
operations model + system API keys to perform translations):

```bash
# e.g. PR #9370 → stage 'pr9370'
./for-env dev pnpm sst shell --stage pr9370 -- \
  pnpm --filter @bike4mind/scripts translations:sync
```

If you need to target a database that isn't the linked resource (e.g. a local
MongoDB), set `MONGODB_URI` explicitly and it takes precedence over `Resource`.

The LLM API keys are read from the database (system keys) by `OperationsModelService`, so no `OPENAI_API_KEY` env var is required.

## Impact

- **English users (95%+)**: Zero HTTP request delay, instant translations
- **Non-English users**: On-demand loading from `public/locales/` (no change)
- **Bundle size**: +13-15KB gzipped (acceptable trade-off for instant English)
