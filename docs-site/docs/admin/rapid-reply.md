# Rapid Reply

The Rapid Reply tab manages a feature that provides users with fast, lightweight AI acknowledgments while their full response is being generated by a more capable model. A smaller "rapid" model sends an instant reply (e.g., "Working through that Python code now!") to reduce perceived latency, and the full response replaces or follows it once ready.

## Header

The page header shows:

- **Title** -- "Rapid Reply Configuration" with a bolt icon
- **Active Chip** -- A green "Active" chip when the feature is globally enabled
- **Refresh Button** -- Reloads the entire page

## Sub-Tabs

The configuration is organized into three tabs:

| Tab | Icon | Description |
|-----|------|-------------|
| Models | Psychology | Model mappings and general settings |
| Metrics | Speed | Performance thresholds and live metrics (auto-refreshes every 30 seconds) |
| Test | Play | Test configuration against specific models |

Badge counts appear on the Models tab showing the number of configured mappings.

## Models Tab

### General Settings

The general settings card includes:

| Setting | Description |
|---------|-------------|
| Feature Status | Global enable/disable toggle for the entire Rapid Reply system |
| Transition Mode | How the rapid reply transitions to the full response. Options include values from `RapidReplyTransitionModes` (e.g., "replace"). The "enhance" mode is currently disabled |
| Fallback Behavior | What happens when the rapid reply fails. Options include values from `RapidReplyFallbackBehaviors` (e.g., "continue"). This field is currently read-only |

A **Save Settings** button persists general setting changes. It is disabled when no changes have been made.

### Model Mappings

Model mappings define which rapid model responds for each main model. The mappings table displays:

| Column | Description |
|--------|-------------|
| Main Model | The primary (slower, more capable) model |
| Rapid Model | The fast model that provides the instant acknowledgment |
| Fallback Priority | Numeric priority for fallback ordering (1-100) |
| Max Tokens | Maximum token count for the rapid reply (10-150) |
| Response Style | The reply style (e.g., "auto", or other values from `RapidReplyResponseStylesCommon`) |
| Max Latency Before Timeout (ms) | Maximum time allowed for the rapid reply before it is abandoned (500-5000ms) |
| Status | Active or Disabled |
| Actions | Edit and Delete buttons |

#### Creating and Editing Mappings

The mapping edit modal provides:

- **Main Model** -- Select from all available models
- **Rapid Model** -- Select from fast/mini models (filtered to names containing "mini", "nano", "haiku", "flash", "small", or "fast"). Each option shows a performance chip based on actual latency metrics when available
- **System Prompt** -- Custom prompt for the rapid reply. Defaults to a template that instructs the model to provide a single-sentence acknowledgment referencing the topic
- **Max Tokens** -- Configurable between 10 and 150
- **Response Style** -- Dropdown of available response styles
- **Max Latency Before Timeout** -- Configurable between 500ms and 5000ms
- **Fallback Priority** -- Numeric priority between 1 and 100
- **Status** -- Enable/disable toggle

#### Deleting Mappings

Deleting a mapping requires confirmation via a browser dialog before removal.

## Metrics Tab

### Performance Thresholds

Configurable thresholds that define acceptable performance:

| Threshold | Range | Default | Description |
|-----------|-------|---------|-------------|
| Max Acceptable Latency | 500-5000ms | 2000ms | Maximum latency before a rapid reply is considered slow |
| Min Success Rate | 0-100% | 90% | Minimum acceptable success rate for rapid replies |

A **Save Thresholds** button persists threshold changes.

### Metrics Display

The RapidReplyMetrics component renders live performance data that auto-refreshes every 30 seconds. Metrics are compared against the configured thresholds to highlight models or mappings that fall outside acceptable parameters.

## Test Tab

### Test Configuration

The test panel allows administrators to verify rapid reply behavior:

1. **Select Model to Test** -- Choose a main model from the dropdown
2. **Test Input** -- Enter a custom message or select from quick test messages:
   - "Hello, how are you?"
   - "Explain photosynthesis in simple terms"
   - "Write a short poem about technology"
   - "What is the capital of France?"
   - "Help me debug this code..."
3. **Run Test** -- Executes the test and displays results
4. **Clear History** -- Removes previous test results

### Test Results

After running a test, results display in several sections:

- **Quick Summary** -- Rapid reply latency, TTFVT (Time to First Visible Token), and response character count
- **Configuration** -- Main model, rapid model, max tokens, response style, max latency, and system prompt used
- **Performance** -- Total latency, rapid reply latency, TTFVT, latency check pass/fail, tokens used, estimated cost, and timing breakdown
- **Rapid Reply Response** -- The actual test input, response text, response length, and word count
- **Current Settings** -- Global enabled state, transition mode, show indicator flag, fallback behavior, and allowed user tags
- **Applicable Prompts** -- Total, active, and applicable prompts with their domains
- **Recommendations** -- System-generated suggestions for improving configuration

### Test History

The most recent 3 tests are shown in a history panel. Each entry shows the model, timestamp, input preview, and success/failure status. Clicking a history entry reloads its model and input for re-testing.

## Best Practices

- **Start with Fast Models** -- Choose the smallest available models (Haiku, Flash, Mini) as rapid models for the lowest latency
- **Keep Prompts Short** -- The default system prompt is optimized for single-sentence acknowledgments; avoid lengthy custom prompts that increase latency
- **Monitor Metrics** -- Regularly check the Metrics tab to identify mappings that exceed latency or success rate thresholds
- **Test Before Enabling** -- Use the Test tab to verify a mapping produces acceptable results before enabling it for users
- **Set Conservative Latency Limits** -- If a rapid reply takes longer than the main model's first token, it defeats the purpose; keep max latency below 2000ms
