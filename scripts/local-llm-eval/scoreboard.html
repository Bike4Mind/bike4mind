Round 2

Hard — untrusted LLM-output parsing

tavern/generate-quests bug #8022

The real production bug: the endpoint ran JSON.parse() straight on raw LLM output and returned HTTP 500 whenever the model replied in prose instead of JSON. Each model had to write a robust parseQuestResponse() surviving 10 cases — prose, fences, JSON buried in text, a decoy {placeholder} brace, shape validation, and never throwing on malformed input.

The plot twist · difficulty isn't linear

score: Round 1 → Round 2

Five models held a flat 10→10. The story is the two crossings: the fast coder rose as the tiny model fell.

What hard mode revealed

It's not reasoning-vs-direct — it's careful-vs-hasty. The lone discriminator on hard mode was recovering JSON past a decoy {placeholder}. Everyone careful caught it — whether by reasoning (glm, gpt-oss, deepseek) or by disciplined coding (qwen2.5-coder). Only the speed-first qwen3-coder cut that corner.

▲ qwen3-coder: 4 → 9. The "easy" task hid a subtler trap (an object-identity Set bug) than the "hard" one. Difficulty isn't where you'd guess.

▼ VibeThinker-3B: 10 → 4. The 3 GB model kept variable scope straight on simple code but not complex code — it declared a value inside a try and used it outside. That's where parameter count finally bit. The easy task masked the ceiling; the hard task surfaced it.

Method: each model got the identical spec one-shot (temp 0.2) and wrote the function in CommonJS; graded on deep-equality clones (Round 1 also catches input mutation; Round 2 counts any thrown exception as a failure). Reasoning models given a 32k-token budget so a long chain of thought can still finish.
